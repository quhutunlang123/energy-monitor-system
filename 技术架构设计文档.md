# 能源监测平台技术架构设计文档

## 1. 技术架构概述

能源监测平台采用分层架构设计，包括硬件层、数据采集层、数据处理层、服务层和前端展示层。系统通过嵌入式设备采集能源使用数据，经过后端服务处理后，通过前端界面展示给用户。

### 1.1 架构分层

| 层次 | 主要职责 | 技术选型 |
|------|----------|----------|
| **感知层** | 数据生成 | 模拟传感器 |
| **网络层** | 本地网络连接 | 实现模块间通信 |
| **平台层** | 本地服务 | 数据存储、分析和处理，权限管理服务 |
| **应用层** | 用户界面和交互 | Web端，支持多角色访问控制 |

## 2. 模拟设备方案设计

### 2.1 模拟设备概述

由于本项目采用本地部署方案，为了避免硬件设备的依赖，我们设计了一套模拟设备方案，通过软件生成模拟传感器数据，实现完整的能源监测功能。

### 2.2 模拟数据生成方案

#### 2.2.1 数据生成算法

- **基础数据模型**：基于真实家庭能源使用模式设计数据模型
- **波动模拟**：加入随机波动和周期性变化，模拟真实设备使用情况
- **场景模拟**：支持配置不同的使用场景（如工作日、周末、节假日）
- **设备行为模拟**：模拟设备开关和使用习惯对能源消耗的影响

#### 2.2.2 模拟设备类型

| 设备类型 | 模拟参数 | 数据范围 | 波动范围 |
|----------|----------|----------|----------|
| 电力设备 | 电压、电流、功率、用电量 | 电压：220V±10%<br>电流：0-10A<br>功率：0-2200W<br>用电量：累积 | 电压：±5%<br>电流：±10%<br>功率：±15% |
| 燃气设备 | 流量、压力、用气量 | 流量：0-10L/min<br>压力：0.1-0.3MPa<br>用气量：累积 | 流量：±10%<br>压力：±5% |
| 水资源设备 | 流量、水位、水质 | 流量：0-20L/min<br>水位：0-100cm<br>pH值：6.5-8.5<br>浊度：0-50NTU | 流量：±15%<br>水位：±5%<br>pH值：±0.2<br>浊度：±10% |

### 2.3 模拟设备管理

#### 2.3.1 设备配置

- **设备参数配置**：可配置设备的基础参数和波动范围
- **数据生成规则**：可配置数据生成的频率和模式
- **场景配置**：可配置不同场景下的设备行为

#### 2.3.2 设备状态管理

- **在线状态**：模拟设备的在线/离线状态
- **故障模拟**：可模拟设备故障状态和异常数据
- **维护状态**：可模拟设备维护状态

### 2.4 模拟数据传输

- **MQTT协议**：使用MQTT协议模拟设备数据传输
- **数据格式**：与真实设备数据格式保持一致
- **传输频率**：默认5秒/次，可配置

### 2.5 模拟设备优势

- **零硬件成本**：无需购买实际硬件设备
- **易于配置**：通过软件界面即可配置和管理模拟设备
- **灵活可调**：可快速调整数据生成规则和参数
- **无环境依赖**：不受物理环境限制，可在任何地方运行
- **易于扩展**：可快速添加新的模拟设备类型

### 3.4 数据流程设计

#### 3.4.1 数据采集流程

1. **模拟数据生成**：
   - 模拟设备管理服务根据配置的规则生成模拟传感器数据
   - 支持配置数据波动范围和模式
   - 模拟真实设备的使用 patterns（如用电高峰、用水高峰等）

2. **数据传输**：
   - 生成的数据通过MQTT协议传输到数据采集服务
   - 数据采集服务将数据转发到Kafka和RocketMQ消息队列

3. **数据处理**：
   - 数据处理服务从Kafka和RocketMQ消费数据
   - 对数据进行校验、转换和计算
   - 将处理后的数据存储到MySQL、MongoDB和Redis

4. **数据存储**：
   - 实时数据存储到realtime_data表
   - 定期将实时数据聚合后存储到history_data表
   - 热点数据存储到Redis缓存
   - 非结构化数据存储到MongoDB

5. **告警处理**：
   - 告警服务实时监测数据是否超过阈值
   - 当数据超过阈值时，生成告警并存储到alert表
   - 通过WebSocket推送给前端

#### 3.4.2 数据查询流程

1. **前端请求**：
   - 用户通过前端界面发起数据查询请求
   - 请求通过API网关转发到相应的服务

2. **服务处理**：
   - API服务接收请求并进行权限验证
   - 根据请求类型调用相应的业务服务
   - 业务服务从MySQL或Redis获取数据

3. **数据返回**：
   - 服务将数据处理后返回给前端
   - 前端接收到数据后进行渲染和展示

         - "8080:8080"
       networks:
         - energy-monitor-network

         - nacos

       image: energy-monitor/auth:latest
       networks:
         - energy-monitor-network

         - nacos
         - redis

       volumes:
         - mysql-data:/var/lib/mysql
       networks:
         - energy-monitor-network
     
     mongodb:
       image: mongo:6.0
       volumes:
         - mongo-data:/data/db
       networks:
         - energy-monitor-network
     
     redis:
       image: redis:7.0
       networks:
         - energy-monitor-network
     
     # 消息服务
     rocketmq:
       image: rocketmqinc/rocketmq:5.0
       ports:
         - "9876:9876"
         - "10911:10911"
       volumes:
         - rocketmq-data:/home/rocketmq/logs
       networks:
         - energy-monitor-network
     
     # 监控服务
     elasticsearch:
       image: elasticsearch:8.0
       environment:
         - discovery.type=single-node
         - ES_JAVA_OPTS=-Xms1g -Xmx1g
       volumes:
         - es-data:/usr/share/elasticsearch/data
       networks:
         - energy-monitor-network
     
     kibana:
       image: kibana:8.0
       ports:
         - "5601:5601"
       networks:
         - energy-monitor-network
       depends_on:
         - elasticsearch
     
     logstash:
       image: logstash:8.0
       networks:
         - energy-monitor-network
       depends_on:
         - elasticsearch
     
     # 服务注册与配置中心
     nacos:
       image: nacos/nacos-server:2.3.0
       environment:
         - MODE=standalone
       ports:
         - "8848:8848"
       networks:
         - energy-monitor-network
   
   volumes:
     mysql-data:
     mongo-data:
     redis-data:
     rocketmq-data:
     es-data:
   
   networks:
     energy-monitor-network:
       external: true

## 8. 技术难点与解决方案

### 8.1 技术挑战

1. **模拟数据的真实性**：
   - 生成的模拟数据需要接近真实使用情况

2. **本地部署的性能优化**：
   - 本地PC性能有限，需要确保系统流畅运行

3. **数据存储的可靠性**：
   - 本地存储可能面临数据丢失风险

4. **用户界面的美观性**：
   - 需要提供专业、美观的用户界面

5. **权限管理的复杂性**：
   - 实现基于RBAC模型的细粒度权限管理，确保权限分配合理

6. **多角色用户体验**：
   - 不同角色的用户需要不同的界面和功能，确保用户体验一致

7. **微服务架构的复杂性**：
   - 本地部署微服务架构，确保服务间通信和依赖管理正确

8. **Nacos本地运行优化**：
   - 确保Nacos在本地环境稳定运行，资源占用合理

9. **Docker容器化部署优化**：
   - 确保Docker容器在Linux虚拟机上稳定运行，资源分配合理

10. **微服务架构复杂性管理**：
    - 管理多个微服务的依赖关系和通信，确保系统稳定

11. **多组件集成协调**：
    - 协调ELK、MongoDB、RocketMQ等多个组件的配置和运行

### 8.2 解决方案

1. **模拟数据的真实性**：
   - 基于真实家庭能源使用模式设计数据生成算法

2. **本地部署的性能优化**：

   - 降低数据采集频率，默认5秒/次，可根据设备性能调整
   - 使用轻量级组件，避免资源密集型技术

3. **数据存储的可靠性**：
   - 提供定期手动备份功能
